4. Priority Gaps & Recommendations
The following gaps represent the highest-risk items based on the implementation analysis.

Critical — missing functionality
KAN-22 (Notifications): No owner notification on transfer initiation. This is a fraud-prevention requirement. Implement email/SMS dispatch from the POST /api/transfers handler using the stored email field and a notification service.
KAN-11 (Citizen Document Upload): Citizens cannot upload title deeds or mutation documents. A dedicated authenticated upload route is needed. Mutation documents are not yet a supported document type in the schema.
KAN-7 (Buyer Approval): No GET endpoint for buyers to view a pending transfer request, and no distinct approval action separated from NADRA simulation.

High — incomplete implementation
KAN-8 (NADRA Failure Path): The simulate-nadra endpoint always succeeds. A rejection scenario must be handled to satisfy the acceptance criterion that failure stops the transfer.
KAN-20 (Flag Disputed Properties): No PATCH/PUT endpoint to mark an existing parcel as disputed after creation. An admin update endpoint is required.
KAN-16 / KAN-26 (Access Control & Judge Role): A judge role with restricted authenticated access is absent. The public parcel routes expose CNIC data to anonymous callers, directly conflicting with KAN-26. Product owner input is needed to resolve the MVP public-access decision against the access-control requirement.

Medium — partial gaps
KAN-3 (Owner Name): The parcel response does not resolve the owner's full name from the User model. A JOIN or lookup is needed.
KAN-24 (Audit Logging): Only transfer events are on-chain. Login, search, admin creation, and document-download events are not confirmed as audited.
KAN-4 (Jira Data Error): The acceptance criteria block in the Jira export for KAN-4 duplicates KAN-6. This should be corrected in the project tracker for accurate sprint tracking.

Low — non-functional
KAN-25 (Performance): No performance SLAs, MongoDB index definitions, or caching strategies are documented. These should be specified and implemented before any load testing.
